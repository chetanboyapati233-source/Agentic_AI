Week 1 · 5 hrs

What is an LLM, actually?

Goal: understand tokens, context, temperature; run your first few calls.

The 60-second explanation

An LLM is a function that takes text in and produces text out. Internally, it's predicting the next token (word-piece) over and over. That's it. Everything else — chat, summarization, coding, RAG — is just clever ways of feeding it text so the "next token" turns out to be useful.

Analogy: An LLM is autocomplete on steroids. Give it a good prompt and it continues in a way that happens to be useful. Give it a bad prompt and it continues confidently into nonsense.

Tokens

A token is roughly ¾ of a word (or about 4 characters). "GenAI" = 2 tokens. "hello world" = 2 tokens. Every model call has a context window — the max tokens of input + output combined. Gemini 1.5 Flash: 1M input tokens. That's huge, but it's not free — you pay per token.

Temperature

ValueBehaviorUse for
0.0DeterministicExtraction, classification, SQL
0.1–0.3Mostly consistentSummaries, Q&A
0.7–1.0Creative, variedBrainstorming, writing

The Gemini API — chat, streaming, structured output

Goal: comfortable with the SDK surface; know when to use each mode.

The 4 modes you'll use constantly

ModeWhenKey code
Single-shotOne Q → one Amodel.generate_content(prompt)
StreamingLong output, live UIstream=True, iterate chunks
ChatMulti-turn conversationchat = model.start_chat()
Structured JSONExtraction, function-likeresponse_schema={...}

System instruction = the rules

Split your prompt into two parts: system instruction (the rules the model should always follow) and user turn (the actual input). This is the single biggest lever for quality and safety.

model = GenerativeModel(
    "gemini-1.5-flash-002",
    system_instruction=(
        "You are a data engineer's assistant. "
        "Answer concisely. Say 'I don't know' if unsure."
    ),
)

Structured output — the killer feature for DE

Pass a JSON schema and the SDK guarantees valid JSON. No more regex-parsing model output.

from vertexai.generative_models import GenerationConfig

schema = {
  "type": "object",
  "properties": {
    "customer": {"type": "string"},
    "amount":   {"type": "number"},
    "date":     {"type": "string"},
  },
  "required": ["customer", "amount"],
}

resp = model.generate_content(
    "Extract: 'Acme paid $12,500 on 2026-04-10.'",
    generation_config=GenerationConfig(
        response_mime_type="application/json",
        response_schema=schema,
        temperature=0.0,
    ),
)
import json
data = json.loads(resp.text)  # guaranteed valid

Week 3 · 6 hrs

Prompt engineering — 5 patterns that cover 90% of work

Goal: stop guessing at prompts; pick the right pattern for the task.

1. Zero-shot

Just ask. Works for simple, well-known tasks.

"Classify this review as positive, negative, or neutral: '...'"

2. Few-shot (the highest-ROI pattern)

Show 2–5 input→output examples, then ask. This locks in format and disambiguates edge cases without any training.

Examples:
"Loved it" → positive
"Meh, okay" → neutral
"Terrible service" → negative

Classify: "Could be better" →

3. Chain-of-Thought (CoT)

For anything involving math, logic, or multi-step reasoning. Add "Think step by step." — accuracy often jumps 10–30%.

4. Structured output (from Week 2)

Use response_schema. Don't ask the model to "return JSON" and hope.

5. Role prompting

"You are a senior SRE. Diagnose…" — cheap way to set tone and expertise level. Use the system instruction for this.

The 3 rules that always apply

Be specific: "Summarize in 3 bullets, each under 15 words" beats "summarize".
Show, don't tell: One example is worth a paragraph of instructions.
Constrain the output: Length, format, allowed values. The model will happily oblige.

Week 4 · 6 hrs

Embeddings — turning text into numbers

Goal: understand vectors, cosine similarity, and top-k search at a gut level.

The 30-second explanation

An embedding model turns any piece of text into a list of numbers (a vector, e.g., 768 numbers for text-embedding-004). Two texts with similar meaning produce vectors that are close to each other in that 768-dimensional space.

Analogy: Think of it as a semantic GPS. Each sentence gets coordinates. "Dog" and "puppy" end up on the same street. "Dog" and "stock market" end up in different cities.

Cosine similarity — measuring closeness

Don't use Euclidean distance on embeddings. Use cosine similarity: the angle between two vectors, ranging from -1 (opposite) to 1 (identical direction). In practice: normalize first, then dot product.

import numpy as np
def cosine(a, b):
    a = a / np.linalg.norm(a)
    b = b / np.linalg.norm(b)
    return float(a @ b)  # dot product after normalization

RETRIEVAL_DOCUMENT vs RETRIEVAL_QUERY

Google's embedding model takes a task_type hint. Use RETRIEVAL_DOCUMENT when embedding your corpus, RETRIEVAL_QUERY when embedding the user's question. This asymmetric training gives ~5–10% retrieval accuracy boost for free.

from vertexai.language_models import TextEmbeddingModel, TextEmbeddingInput

m = TextEmbeddingModel.from_pretrained("text-embedding-004")
doc_vecs = m.get_embeddings([
    TextEmbeddingInput(text=chunk, task_type="RETRIEVAL_DOCUMENT")
    for chunk in chunks
])

Top-k search

Embed the query. Compute cosine against every doc. Sort. Return top k. This is the whole of "search" at toy scale — and the mechanics don't change when you scale to Vertex AI Vector Search. It just gets faster.

Week 5 · 7 hrs

Basic RAG — the single most important pattern

Goal: build an in-memory RAG end-to-end. This is what interviewers want to see.

RAG in one picture

Ingest docs → chunk into 500–1000 token pieces → embed each chunk → store in an index → at query time, embed the question, retrieve top-k chunks → stuff them into the prompt → generate the answer with grounding.

Mental model: RAG is an open-book exam. You aren't teaching the model new facts — you're just handing it the right page before it answers.

Chunking — the part everyone gets wrong

Three strategies, pick based on your data:

StrategyBest for
Fixed-size (800 tokens, 100 overlap)Generic text, safe default
Sentence-basedShort, clean content (FAQs, product specs)
Markdown-aware (by heading)Docs, wikis — preserves structure

Golden rule: each chunk should make sense on its own. If someone dropped you into the middle, would you know what the chunk is about? If no, add more overlap or chunk by section.

The grounded prompt template

system_instruction = """
You answer questions using ONLY the provided sources.
Cite sources by ID in [brackets].
If the sources don't contain the answer, say:
'I don't know based on the provided sources.'
"""

prompt = f"""
SOURCES:
{top_k_chunks}

QUESTION: {user_question}
"""
Common mistake: putting sources in the system instruction. Don't. System = rules. User = data. Mixing them makes prompt injection easy.

Week 6 · 7 hrs

Production RAG — Vertex AI Vector Search & BigQuery

Goal: scale beyond "fits in memory." Pick the right vector store for DE workloads.

Two paths, same result

OptionWhen to pick it
Vertex AI Vector SearchMillions+ vectors, sub-100ms latency, dedicated endpoint
BigQuery VECTOR_SEARCHVectors live next to your warehouse data, SQL-native, no endpoint

For a data engineer, BigQuery is usually the better starting point: no extra service, no deployed endpoint to forget about, and you can join embeddings to your dimension tables in one query.

BigQuery vector — the essentials

-- 1. Table with an embedding column (REPEATED FLOAT64)
CREATE TABLE mydataset.docs (
  id STRING,
  text STRING,
  embedding ARRAY<FLOAT64>
);

-- 2. Vector index (IVF)
CREATE VECTOR INDEX docs_idx ON mydataset.docs(embedding)
OPTIONS(distance_type='COSINE', index_type='IVF');

-- 3. Query
SELECT base.id, base.text, distance
FROM VECTOR_SEARCH(
  TABLE mydataset.docs, 'embedding',
  (SELECT [0.1, 0.2, ...] AS embedding),
  top_k => 5
);

Vertex AI Vector Search — the essentials

  • Index: the data structure (tree-AH or brute-force).
  • Index endpoint: the serving layer (billed hourly when deployed — clean up at end of day).
  • Deployed index: binding of the two.
Cost trap: A deployed Vector Search endpoint bills by the hour whether you query it or not. Run python 07_rag_vertex_vector_search.py cleanup before bed.

Week 7 · 6 hrs

Function calling — letting the LLM use tools

Goal: understand how LLMs trigger Python functions; build a 3-tool mini-assistant.

The idea

You describe a set of Python functions to Gemini as a schema. When the user asks something that needs one of them (e.g., "what time is it in Tokyo?"), the model returns a structured function call instead of plain text. Your code runs the function, sends back the result, and the model continues.

Analogy: You hire an assistant and hand them a rolodex of people they can call. They can't dial the phone themselves — but when a task needs a phone call, they tell you whom to call and with what message. You dial, read them the result, they continue.

The loop (always the same)

  1. User asks a question.
  2. Model returns either (a) a text answer, or (b) function_call(name, args).
  3. If (b): run the Python function locally with those args.
  4. Send result back to model as function_response.
  5. Go to 2 until (a).

Declaring a tool

from vertexai.generative_models import Tool, FunctionDeclaration

get_weather = FunctionDeclaration(
    name="get_weather",
    description="Get current temperature for a city",
    parameters={
        "type": "object",
        "properties": {"city": {"type": "string"}},
        "required": ["city"],
    },
)
tools = [Tool(function_declarations=[get_weather])]

model = GenerativeModel("gemini-1.5-flash-002", tools=tools)

Week 8 · 6 hrs

Agents — multi-step workflows with LangGraph

Goal: build a graph of LLM + tools + self-correction. The pattern for serious automation.

Agent = function calling in a loop with memory

An agent is basically: plan → act → observe → repeat. LangGraph gives you a state machine where each node can be an LLM call, a tool call, or a decision. You define the edges, it runs the loop.

The 4-node pattern you'll reuse forever

PLAN → WRITE_SQL → EXECUTE → SUMMARIZE
                        ↑         ↓
                        └── retry if error ──┘  (max 3 times)

Always guard the loop

Rule: every agent must have a MAX_ITER. Otherwise it can spin forever and burn your budget. Start with 3. Only raise it if you have a good reason.

State — the one thing to get right

Your state object flows through every node. Keep it small and typed (TypedDict):

class AgentState(TypedDict):
    question: str
    sql: str | None
    rows: list | None
    error: str | None
    iter: int
    answer: str | None

Week 9 · 6 hrs

Evaluation — treating LLMs like software

Goal: build a golden set + judge. Know quality before you ship.

Evals are unit tests for LLMs

If you don't measure it, you can't improve it. Build a golden set of 20–50 Q&A pairs, commit it to git, and run evals on every change. Track scores over time like you track build pass-rates.

The 4 metrics you'll actually use

MetricWhat it checksLimit
Exact-matchpred == refOnly works for extraction
ROUGE-LLongest common subsequence of wordsWord-level, misses synonyms
Embedding cosineSemantic similarity of pred vs refSmooth, robust to paraphrase
LLM-as-judgePro model scores 1–5 with rubricExpensive; use sparingly

LLM-as-judge, done right

  • Use a stronger model than the one you're testing (e.g., Gemini Pro judges Flash).
  • Give it a rubric with criteria: factuality, completeness, grounding.
  • Use response_schema so scores come out as integers, not prose.
  • Temperature 0.0 for reproducibility.

Week 10 · 5 hrs

Fine-tuning — usually not what you need

Goal: know the 3 cases where SFT actually helps; run one real job.

The honest truth about fine-tuning

90% of the time, prompting + RAG wins. Fine-tuning is slow to iterate, costs money, and can make the model worse if your data is mediocre.

The 3 cases where SFT is right

  1. Style / tone — you need a consistent voice prompting can't lock in.
  2. Strict output format — a schema so rigid you need thousands of examples.
  3. Domain vocabulary — the model has never seen your jargon.
Fine-tuning does NOT add new facts. Those still come from the prompt or RAG. If you want the model to know your company's docs, use RAG. Period.

Vertex AI SFT format

{"contents": [
  {"role": "user",  "parts": [{"text": "Kafka lag on 'orders' topic"}]},
  {"role": "model", "parts": [{"text": "{\"service\":\"orders\",\"sev\":4,...}"}]}
]}

One JSON object per line, upload to GCS, point sft.train() at it.

Cost: $10–$50 per tuning run. Don't fire it "just to see." Prepare ~50 examples first, run once, evaluate rigorously.

Week 11 · 6 hrs

Safety and Serving — what makes it "production"

Goal: PII, prompt injection, cost caps, FastAPI endpoint.

The 5 safety concerns, in priority order

  1. PII leakage — scrub before calling LLM with Cloud DLP.
  2. Prompt injection — attackers hide instructions in input. Separate trust zones; use Model Armor.
  3. Unsafe SQL — LLM-written SQL must be allow-listed to SELECT only. Cap bytes scanned.
  4. Cost blow-up — per-user rate limits; budget alerts; default to Flash.
  5. Hallucination — reduce with grounding (RAG) + "I don't know" prompt instruction.

SQL allow-list (copy this pattern)

SQL_ALLOWED = re.compile(r"^\s*select\s", re.IGNORECASE)
SQL_BANNED  = re.compile(r"\b(delete|update|insert|drop|truncate|create|alter|merge)\b",
                         re.IGNORECASE)

def is_sql_safe(sql):
    if SQL_BANNED.search(sql):  return False, "Only SELECT allowed."
    if not SQL_ALLOWED.match(sql): return False, "Must start with SELECT."
    return True, ""

FastAPI in 15 lines

from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class AskBody(BaseModel):
    question: str

@app.post("/ask")
def ask_endpoint(body: AskBody):
    return ask(body.question).model_dump()

@app.get("/health")
def health():
    return {"ok": True}

Observability — the 4 metrics to log

Latency p50/p95, tokens in + out, error rate, cost per 1k requests. Dump to Cloud Logging; chart in Cloud Monitoring.

Week 12 · 8 hrs

Capstone — Data Engineer Assistant

Goal: one portfolio project that demonstrates every concept from W1–11.

What you're building

A natural-language interface over your BigQuery warehouse. User asks "How many orders per day last week?" — the system:

  1. Retrieves relevant schema & docs (W5–6 RAG)
  2. Writes a SQL query (W7 function calling)
  3. Validates it against the allow-list (W11 safety)
  4. Executes and summarizes the result (W3 prompt engineering)
  5. Serves it over a FastAPI endpoint (W11)
  6. Is measured against a golden eval set (W9)

The shipping checklist

  • End-to-end /ask endpoint works for 5 distinct questions.
  • SQL allow-list blocks DELETE, UPDATE, etc.
  • Eval harness runs on a 10-item golden set; mean judge score ≥ 4.
  • 5-minute demo video recorded (screen + voice).
  • GitHub README: problem, architecture diagram, how to run, results table.
  • Push to github.com/<you>/data-engineer-assistant.

The "walk me through a project" script

Every interview will ask this. Rehearse a 3-minute version of:

  1. Problem: "Analysts at my company had to wait days for simple SQL. I built…"
  2. Architecture: Draw the RAG + agent + safety flow on a whiteboard.
  3. Trade-offs: "I picked BigQuery over Vector Search because…"
  4. Results: "On my golden set, mean judge score was 4.2/5, p95 latency 3.1s."
  5. What I'd do next: Two concrete improvements.
